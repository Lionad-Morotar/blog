# Step 10: 多受众翻译器

## 目标

帮助用户将资料的核心洞见转化为不同受众能理解的内容，实现有效传播。

## 何时执行

**必须执行的情况：**
- 用户需要向不同背景的人汇报/解释资料内容
- 资料将用于跨部门/跨层级沟通
- 需要撰写不同版本的材料（内部/对外）
- 用户问"怎么向 X 解释这个概念"

**核心价值：**
消除专业壁垒，让知识在不同语境中流动。

## 执行流程

### 1. 受众分析

首先明确需要覆盖的受众类型：

**常见受众分类：**

| 受众类型 | 核心关切 | 专业深度 | 决策影响力 |
|---------|---------|---------|-----------|
| 高管/决策者 | ROI、战略影响、风险 | 低（广度优先） | 高 |
| 技术/专业 | 实现细节、技术可行性 | 高（深度优先） | 中 |
| 业务/运营 | 实施难度、资源需求 | 中 | 中 |
| 最终用户 | 对我有什么好处 | 低 | 低 |
| 外部/公众 | 可信度、相关性 | 极低 | 视情况而定 |

**分析维度：**
- 他们的知识背景是什么？
- 他们最关心什么？
- 他们有多少时间？
- 什么会让他们产生抵触？
- 什么能说服他们？

### 2. 核心信息提取

确定所有受众版本都必须包含的核心信息：

- 最关键的结论（如果只记得一点，应该是什么）
- 必须支持这个结论的关键证据
- 最重要的行动建议

### 3. 分受众转化

为每个受众类型定制内容：

#### A. 语言层面

| 受众 | 语言风格 | 术语使用 | 举例 |
|-----|---------|---------|------|
| 高管 | 简洁有力、结果导向 | 避免技术术语 | "提升效率"而非"优化算法" |
| 技术 | 精确、结构化 | 专业术语+解释 | "使用 Transformer 架构（一种注意力机制）" |
| 业务 | 实用、场景化 | 业务术语 | "缩短交付周期" |
| 用户 | 易懂、有感染力 | 生活化比喻 | "像有个智能助手" |

#### B. 结构层面

**高管版本：**
- 结构：结论先行 → 关键论据 → 资源需求 → 下一步
- 长度：1 页以内
- 重点：Why（为什么做）和 What（做什么）

**技术版本：**
- 结构：问题定义 → 技术方案 → 实现细节 → 验证结果
- 长度：详细文档
- 重点：How（怎么做）和 Trade-offs（取舍）

**业务版本：**
- 结构：场景 → 解决方案 → 实施步骤 → 预期效果
- 长度：适中（3-5 页）
- 重点：Who（谁来做）和 When（何时完成）

**用户版本：**
- 结构：痛点 → 价值主张 → 使用方式 → 行动号召
- 长度：简洁（1 页或更少）
- 重点：Benefit（对我有什么好处）

#### C. 案例/举例

为每个受众提供他们能理解的例子：
- 高管：商业案例、财务影响
- 技术：代码示例、架构图
- 业务：流程改进前后对比
- 用户：日常使用场景

### 4. 冲突管理

识别不同受众之间可能的冲突：

- 高管关注的 ROI vs 技术关注的完美实现
- 业务关注的速度 vs 技术关注的质量
- 用户关注的便利 vs 公司关注的成本

**处理策略：**
- 在各版本中预置回应其他受众关切的内容
- 提供协调不同需求的框架

### 5. 传播媒介建议

推荐适合各受众的传播形式：

| 受众 | 推荐媒介 | 备选媒介 |
|-----|---------|---------|
| 高管 | PPT 演示、一页纸摘要 | 邮件简报 |
| 技术 | 技术文档、代码仓库 | 技术分享会 |
| 业务 | 流程图、SOP 文档 | 培训工作坊 |
| 用户 | 产品内引导、FAQ | 视频教程 |

## 提示词

**中文（替换受众）：**
```
将这些资料的见解转化为三种不同受众群体能够理解的内容：[高管、工程师、最终用户]。针对每种受众群体，重点关注他们具体关心的内容，并使用他们能够立即理解的语言/示例。

输出要求：
1. 提炼核心信息（所有版本共通的一句话结论、关键证据、行动建议）
2. 每个受众一个完整版本，包含：
   - 受众特征分析（时间、关切、背景）
   - 定制化的内容结构
   - 合适的语言风格
   - 相关的案例/比喻
3. 版本对照表（方便快速对比）
4. 冲突协调建议（不同受众之间的潜在张力及处理方式）
5. 传播建议（媒介、时机、注意事项）
```

**English（替换受众）：**
```
Translate the insights from these sources for three different audiences: [executives, engineers, end-users]. For each audience, focus on what they specifically care about and use language/examples they'll immediately understand.

Output requirements:
1. Extract core message (common across all versions: one-sentence conclusion, key evidence, action recommendation)
2. Complete version for each audience, including:
   - Audience profile (time budget, concerns, background)
   - Customized content structure
   - Appropriate language style
   - Relevant cases/metaphors
3. Version comparison table (for quick reference)
4. Conflict mediation suggestions (potential tensions between audiences and how to address)
5. Communication recommendations (medium, timing, notes)
```

## 执行检查清单

- [ ] 是否分析了各受众的特征？
- [ ] 核心信息是否提炼准确？
- [ ] 每个版本是否针对受众关切定制？
- [ ] 语言风格是否与受众匹配？
- [ ] 是否提供了相关的案例/比喻？
- [ ] 长度是否适合受众的时间预算？
- [ ] 是否识别了受众间的潜在冲突？
- [ ] 传播建议是否实用？

## 注意事项

- **避免过度简化**：简化不等于失真，确保核心信息准确
- **尊重受众智慧**：不要居高临下，即使是用户版也要尊重
- **保持一致性**：不同版本的事实要一致，只是表述不同
- **测试反馈**：如有可能，让目标受众预览并收集反馈

## 示例对比

**同一信息的不同表达：**

- **技术原文**："我们使用 Transformer 架构的自注意力机制处理序列数据，实现了 O(1) 的并行复杂度"
- **高管版**："新算法将数据处理速度提升 10 倍，缩短项目交付周期"
- **业务版**："采用最新 NLP 技术，能同时处理大量文档，无需按顺序等待"
- **用户版**："你的问题会立即得到回复，不用排队等待"
