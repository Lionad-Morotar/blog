# Archive

## Overview

将已整理好的内容归档到用户的笔记系统中。工作流程包含：调研笔记系统结构、制定归档计划、用户确认、执行归档和提交。

## Workflow

### Step 1: 调研笔记系统

探索用户的笔记系统结构：

1. 首先读取 `content/6.maps/Agents.md` 获取笔记系统规范（如有）
2. 使用 Glob 查找笔记目录（常见模式：`content/**/maps/**/*`, `**/*.md`）
3. 读取关键索引文件（如 `0.index.md`, `index.md`, `<dirname>/<dirname>.md`）
4. 使用 Grep 搜索与待归档内容相关的关键词，了解类似资料在现有笔记系统中的分类方式
5. 分析现有分类结构和文件命名规范
6. 确定知识点的最佳归档位置

**参考文档：**
- `content/6.maps/Agents.md` - 笔记系统智能体档案，包含目录结构和格式规范

**常见笔记系统结构：**
- `content/6.maps/_topic/subtopic/file.md` - 三级分类
- `content/notes/` - 扁平或层级结构
- `docs/knowledge-base/` - 文档型结构

### Step 2: 制定归档计划

基于调研结果制定计划：

1. 确定每个知识点的归档位置（文件路径）
2. 确定归档格式（新增小节 vs 新建文件）
3. 检查是否需要更新索引
4. 用 mermaid 图表展示计划结构

**计划模板：**
```markdown
## 归档计划

| 知识点 | 归档位置 | 操作 |
|--------|----------|------|
| [标题] | `[路径]` | 新增/修改 |

### 文件变更清单
- [ ] `[文件1]` - [改动描述]
- [ ] `[文件2]` - [改动描述]
```

### Step 3: 等待用户确认

**必须等待用户明确确认后才能执行。**

向用户展示：
1. 待归档内容摘要
2. 归档计划（包含路径和操作）
3. 询问是否执行

**确认话术：**
"以上是将内容归档到笔记系统的计划，确认后我将执行归档并提交。"

### Step 4: 执行归档

用户确认后：

1. 按需编辑或新建 markdown 文件
2. 更新相关索引文件
3. 保持与现有笔记格式一致
4. **添加引用**：在相关索引文件或关联文档中添加对相关文件的引用，便于导航发现

### Step 5: 提交更改

归档完成后：

1. `git add` 相关文件
2. `git commit` 使用清晰的提交信息

**注意**：默认只提交到本地，不自动推送到远程。如需推送，请用户手动执行 `git push` 或使用 git-push 技能。

## 格式规范

### Frontmatter 格式
```yaml
---
title: [标题]
description: [简介]
---
```

### 知识点格式
- 使用 `####` 作为知识点标题
- 内容简洁，保留关键信息
- 知识点之间**不使用 `---` 分隔线**，直接连续排列
- 包含链接，**链接数量规则**：
  - 普通知识点：无需链接
  - 一般知识点：1 个链接
  - 少数情况（需对比不同视角）：2 个或多个不同方向的链接
  - 常识性概念可以包含在其他知识点中一笔带过

### 人物笔记格式
```yaml
---
title: [姓名]
description: [身份/职业]
---

#### 简介？

[简介内容]

#### 主要贡献/观点？

[要点]

见：[链接](URL)
```

## 注意事项

- **质量优于速度**：充分调研笔记系统结构，确保归档位置合理
- **格式一致性**：遵循目标笔记系统的现有格式
- **索引更新**：新建文件时必须更新索引
- **原子提交**：一次归档一个主题，便于追溯
