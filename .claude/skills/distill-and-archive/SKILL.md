# Distill And Archive

## Overview

将网络文章或内容提炼为精炼知识点，并归档到用户的笔记系统中。工作流程包含：内容提取与 distill、笔记系统调研、归档计划制定、用户确认、执行归档和提交。

## 归档模式

### 模式一：简单归档

当用户明确要求"简单归档"时，采用精简流程：

**适用场景：**
- 内容已在笔记系统中存在合适位置（如链接列表、索引文件）
- 只需添加链接引用，无需 distill 详细知识点
- 快速整理多个相关网址

**流程：**
1. **调研笔记系统** - 按照**四层认知结构**确定归档位置（Domain → Subdomain → Topic）
2. **检查现有内容** - 确认哪些 URL 已存在、哪些需要添加
3. **展示计划** - 简要列出需要添加的链接及归档位置（包含四层认知结构分析）
4. **执行归档** - 添加链接到合适位置
5. **关闭标签**（如用户要求）

**归档格式：**
```markdown
* [文章标题](URL): 简短描述（可选）
```

**格式要点：**
- **仅添加链接**，不需要 `####` 小标题
- **不需要详细内容介绍**，将内容简述放入链接说明中
- **避免使用笼统的小节标题**（如"## 学习资源"、"## 参考链接"），应将链接归入现有相关小节或创建有具体含义的小节
- 保持与目标文件中现有链接格式一致
- **必须遵循四层认知结构**：即使是简单归档，也要按照 Domain → Subdomain → Topic 的层级找到合适的归档位置，而不是随意选择文件

### 模式二：完整归档（默认）

标准工作流程，见下文详细步骤。

## Workflow

### Step 1: Distill 内容

当用户提供 URL 时：

1. 使用 web-reader 工具获取文章内容
2. **搜索补充与验证**（避免"白大褂陷阱"）：
   - **优先查找一手来源**：如文章引用论文/报告，优先搜索原文而非二手解读
   - **区分事实与观点**：对关键数据/主张，搜索是否有权威来源交叉验证
   - **警惕"流畅性偏见"**：不因为某来源写得通俗易懂就优先采用
   - **控制来源数量**：2-3 个高质量来源优于 10 个低质量来源
3. 将内容整理为精炼的知识点（takeaways）

**知识组织原则：**
- **简单知识**（围绕单一核心观点）：保持为一个知识点，不强行拆分
- **复杂知识**（多个独立论点或视角）：拆分为若干关联知识点
- 判断标准：知识点之间是否必须相互引用才能理解？如果是，则属于同一知识单元

**知识、观点、洞见：**
- **知识（Knowledge）**：事实性、客观、可验证的内容
- **观点（Opinion）**：作者的主观经验、实践偏好、流派主张
- **高层次洞见（Insight、元思考、元认知）：**：部分高难度网站或信息特殊的网站，可提炼高层次洞见——捕获思维模式而非操作细节。这种洞见关注作者如何理解问题、建立概念关联、形成独特的认知框架，而非具体的技术步骤或实现方法。
- **处理方式**：采用混合归档，统一使用 `####` 标题格式，通过措辞体现区别：
  - 知识：直接陈述事实（"XX 是..."、"XX 的工作原理是..."）
  - 观点：使用体现主观性的措辞（"XX 建议..."、"XX 的实践是..."、"一种观点认为..."）

**输出格式：**
```markdown
#### [标题]

[内容：使用朴素的单词，精确术语，符合认知习惯，融入轻微情感表达]

见：[链接标题](链接地址)
```

### Step 2: 调研笔记系统

探索用户的笔记系统结构，按照**四层认知结构**确定归档位置。

#### 2.1 快速预览文件内容（推荐）

在读取完整文件前，**优先使用脚本提取文件元数据**，避免大文件撑满上下文：

```bash
node .claude/skills/distill-and-archive/scripts/extract-markdown-meta.js <文件路径> [--depth=3]
```

脚本返回 [TOON 格式](https://toonformat.dev/):
```
file: filename.md
hasFrontmatter: true
title: 标题
description: 描述...
toc[N]{level,text,anchor}:
  1,标题1,anchor-1
  2,标题2,anchor-2
totalHeadings: N
```

**使用策略:**
- ✅ **优先使用脚本** - 快速判断文件结构,避免加载完整内容
- ✅ **根据 TOC 决策** - 通过标题层级判断知识点归属
- ⚠️ **Fallback 到 Read** - 仅当标题信息不足时才读取全文

#### 2.2 四层认知结构（Domain → Subdomain → Topic → Knowledge Point）

| 层级 | 定义 | 物理形式 | 判断标准 |
|------|------|----------|----------|
| **领域 (Domain)** | 最高层级的知识边界 | `_domain/` 目录 | 存在于 `0.index.md` 的导航结构中，目录以 `_` 开头 |
| **子领域 (Subdomain)** | 领域内的专业分支 | `subdomain/` 子目录或文件 | 构成独立学习单元，有明确知识边界 |
| **主题 (Topic)** | 具体的技术点或概念 | `.md` 文件 | 超过 150 行或有扩展潜力的内容拆分为独立文件 |
| **知识点 (Knowledge Point)** | 原子化的思考性内容 | `####` 四级标题 | 观点、案例、洞见，不拆分为独立文件 |

**调研步骤：**

1. 使用 Glob 查找笔记目录（常见模式：`content/**/maps/**/*`, `**/*.md`）
2. **优先使用脚本预览关键索引文件**（如 `0.index.md`, `index.md`）获取结构概览，避免直接读取大文件
3. 使用 Grep 搜索与待归档内容相关的关键词，了解类似资料在现有笔记系统中的分类方式
4. **置信度中等**：读取 `content/6.maps/Agents.md` 获取笔记系统规范作为参考（如文件存在）
5. **分析内容本质，按四层认知结构确定归档位置：**
   - **领域判断**：内容属于哪个宏观领域？（前端、工程、数据库、商业等）
   - **子领域判断**：是否构成独立学习单元？（如 CSS、JavaScript、Docker 等）
   - **主题判断**：是否超过 150 行或有独立扩展潜力？
   - **知识点判断**：内容是事实描述还是思考性洞见？
6. **多个知识点的处理**：一个网站可能 distill 出多个独立知识点，需**分别确定归档位置**
   - 判断标准：知识点之间是否必须相互引用才能理解？如果不是，则属于不同知识单元
   - 每个知识点独立分析其领域-子领域-主题层级
7. 现有项目结构仅作为参考（50%权重），**认知结构优先**于现有文件布局

**参考文档：**
- `content/6.maps/Agents.md` - 笔记系统智能体档案，包含目录结构和格式规范（置信度中，作为辅助参考）
- `.planning/taxonomy-criteria.md` - 四层认知结构的详细判断标准

**常见笔记系统结构：**
- `content/6.maps/_domain/subdomain/file.md` - 知识地图（四层结构）
- `content/7.tools/*.md` - 工具收集（独立目录）
- `content/notes/` - 扁平或层级结构
- `docs/knowledge-base/` - 文档型结构

### Step 3: 更新 Agents.md（如需要）

在制定归档计划前，检查是否需要更新 `content/6.maps/Agents.md`：

**触发条件：**
- 发现新的分类目录未在 Agents.md 中记录
- 归档位置涉及 Agents.md 未列出的领域或子领域
- 笔记系统结构有较大变化

**更新内容：**
- 补充新发现的分类目录到"笔记系统结构"部分
- 如有新的格式规范，添加到"文件格式规范"部分
- 保持 Agents.md 作为笔记系统的活文档

**注意：** 简单归档模式下可跳过此步骤。

### Step 4: 制定归档计划

基于调研结果制定计划：

1. **进行四层认知结构分析** —— 对每个知识点，明确其在 Domain → Subdomain → Topic → Knowledge Point 四层结构中的位置
2. 确定每个知识点的归档位置（文件路径）
3. 确定归档格式（新增小节 vs 新建文件）
4. 检查是否需要更新索引
5. 用列表展示计划结构

**归档计划完整模板：**
```markdown
## 归档计划

### 四层认知结构分析

| 层级 | 判断 | 说明 |
|------|------|------|
| **领域 (Domain)** | `_ai` | AI 技术与应用 |
| **子领域 (Subdomain)** | `agents` | Agent 架构与设计 |
| **主题 (Topic)** | `multi-agent-frameworks.md` | 多智能体框架对比 |
| **知识点 (Knowledge Point)** | `####` 四级标题 | 新增：架构选择决策框架 |

### 建议归档位置

将内容作为新的知识点添加到 `[文件路径]` 文件中，新增 `[小节名称]` 小节。

**理由示例**
- 该文件已包含相关内容，与待归档主题高度相关
- 内容是对现有主题的补充/延伸
- 符合知识点格式（原子化思考性内容，使用 `####` 标题）

### 文件变更清单
- [ ] `[文件1]` - [改动描述]
- [ ] `[文件2]` - [改动描述]
```

### Step 5: 确认与执行

**模式一：简单归档（自动执行）**

当用户明确要求"简单归档"时，跳过确认步骤，直接执行：
1. **进行四层认知结构分析** - 确定内容在 Domain → Subdomain → Topic 中的位置
2. 简要展示归档计划（包含四层结构分析 + 位置 + 链接）
3. 立即执行归档
4. 提交更改
5. 向用户报告结果

**模式二：完整归档（需确认）**

标准模式下，**必须等待用户明确确认后才能执行**。

向用户展示：
1. Distill 后的知识点
2. 归档计划（包含路径和操作）
3. 询问是否执行

**确认话术：**
"以上是将知识点归档到笔记系统的计划，确认后我将执行归档并提交。"

### Step 6: 等待确认，执行归档

用户确认后：

1. 使用 TodoWrite 跟踪进度
2. 按需编辑或新建 markdown 文件
3. 更新相关索引文件
4. 保持与现有笔记格式一致
5. **添加引用**：在相关索引文件或关联文档中添加对相关文件的引用，便于导航发现

### Step 7: 提交更改

归档完成后：

1. `git add` 相关文件
2. `git commit` 使用简洁的提交信息

**提交信息格式（优先使用简单格式）**：

```
gists: 补充/更新/删除 [主题/文件] [简要描述]
```

**示例**：
- `gists: 补充智能健身教练的快慢思考分析`
- `gists: 更新多智能体框架对比内容`
- `gists: 添加 Workflow vs Agent 对比案例`

**注意**：
- 默认只提交到本地，不自动推送到远程
- 如需推送，请用户手动执行 `git push` 或使用 git-push 技能

## 格式规范

### Frontmatter 格式
```yaml
---
title: [标题]
description: [简介]
---
```

### 知识点格式
沿用 distill-takeaways 的输出风格：
- 使用 `####` 作为知识点标题
- 内容简洁，保留关键信息
- 知识点之间**不使用 `---` 分隔线**，直接连续排列
- **行长度控制**：单行不要太长，中文最长 120 字左右，不然拆成两个句子，在语义完整处（如逗号、句号后）断行，提升可读性和 diff 友好性
- 包含链接，**链接数量规则**：
  - 普通知识点：无需链接
  - 一般知识点：1 个链接
  - 少数情况（需对比不同视角）：2 个或多个不同方向的链接
  - 常识性概念可以包含在其他知识点中一笔带过
  - 如果链接和知识点只是拓展关系，可选补充链接说明

### 人物笔记格式
```yaml
---
title: [姓名]
description: [身份/职业]
---

#### 简介？

[简介内容]

#### 主要贡献/观点？

[要点]

[链接格式]
```

## 链接格式

```md
见：[链接](URL)、[链接2](URL)：链接2说明
```

## 注意事项

- **质量优于速度**：Distill 时充分搜索，确保信息准确
- **格式一致性**：遵循目标笔记系统的现有格式
- **索引更新**：新建文件时必须更新索引
- **原子提交**：一次归档一个主题，便于追溯
- **避免搜索陷阱**：
  - 带引用的内容 ≠ 可信，点击链接验证其实际内容
  - 写得流畅易懂的来源 ≠ 权威，优先官方文档/学术来源
  - AI/内容农场内容可能互相引用形成"衔尾蛇"，注意识别
