# Lionad 博客风格 —— 操作指令

## 风格定位

第一人称技术随笔。以个人排查/思考过程为主线，把一件事讲清楚，但不求面面俱到。

## 核心原则

1. **大幅删减，只留主线**  
   删除支线、过度环境交代、完整命令输出、发散分析。保留：我遇到了什么 → 我第一反应是什么 → 试了什么 → 卡在哪 → 真正原因是什么 → 怎么解决的。

2. **口语化，但不过度口水**  
   像口述一样自然，保留短句、断句和少量情绪词。避免“综上所述”“下面我们来分析”“先交代环境”等报告体表达，也避免“嘿”“啊”等过于口水的感叹词。

3. **一个核心点讲透**  
   技术解释不贪多。从问题链中选出最关键的因果环节，把它讲清楚，其他点到为止或省略。

4. **问题导向的小标题**  
   用疑问句或短语推进节奏。避免“环境”“验证”“总结”这类文档式标题。

5. **只留关键片段**  
   代码、日志、数据只保留最能说明问题和方案的关键帧，不铺完整输出。

## 改写动作

### 必做

- [ ] 删除 30% 以上非主线内容
- [ ] 把报告体/说明体改为第一人称口述
- [ ] 收敛技术解释到一个核心因果点
- [ ] 用小标题推进，一段一事
- [ ] 保留关键报错/配置/验证片段

### 不做

- [ ] 改得像教程或官方文档
- [ ] 罗列所有可能原因
- [ ] 用“先交代环境”“对照实验”等正式表述
- [ ] 删除所有个人语气和情绪
- [ ] 添加“综上所述”式总结

## 语气参考

**可用**：
- 今天打开 xx 时...
- 整整一排 / 一连串
- 这就奇怪了
- 直觉告诉我
- 第一反应是
- 这里就触及到知识盲区了
- OK，搞定了

**避免**：
- 先交代环境，后面所有判断都依赖它
- 下面我们来看
- 对照实验
- 综上所述
- 由此可见

## 通用结构模板

```markdown
## 现象
（简短场景 + 关键错误/输出）

## 是 xx 问题吗？
（提出猜想 + 一句话验证）

## 怎么修改呢？
（第一次尝试 + 失败/卡点）

## 隐藏机制：xxx
（核心根因，讲透一个点）

## 最终方案：xxx
（配置/代码/命令 + 验证）
```

## 自检清单

- [ ] 能否用 3 句话讲完这次经历的主线？
- [ ] 是否只深入讲了一个核心机制/原因？
- [ ] 是否保留了至少一处第一人称的口语化表达？
- [ ] 是否删除了大量非必要的环境铺垫和完整输出？
- [ ] 代码/数据片段是否只保留了关键帧？
- [ ] 是否有“综上所述”式的总结段落？（应该有 = 不通过）
