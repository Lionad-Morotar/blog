# Lionad 博客风格 —— 完整拆解

## 一句话定义

**用第一人称口述的节奏，把一次真实技术经历的主线讲清楚，只抓一个核心点深入，不贪全。**

## 核心公式

```
我遇到了什么问题 → 我第一反应是什么 → 我试了什么 →
卡住的真正原因是什么（讲透一个点）→ 最后怎么解决的
```

## 六维特征

### 1. 篇幅：精简到骨架

删除环境铺垫、完整命令输出、支线分析、过度论证。每个章节 1-3 段，一段一事。

**对比示例**：

> 原文：该项目是一个基于 Vue 2 的老后台系统，使用 webpack 4 构建，团队历史原因导致 Node 版本长期锁定在 16。本机通过 fnm 管理 Node，default 为 22，
但项目根目录的 `.node-version` 文件会在进入目录时自动切换版本...
>
> 改写：因为在一些老项目，我通过 fnm 维护项目级别的 Node.js 版本：全局默认 22，项目根 `.node-version` 锁定 16。

### 2. 语气：第一人称口述

保留“我”的视角，允许短句和口语连接词，情绪词作为节奏标点，但不过度口水。

**对比示例**：

> 原文：四个服务同时失败，涉及三种不同的启动方式，且均报错 Connection closed，但其余服务运行正常。
>
> 改写：重新启动也还是这 4 个报错，而且诡异的是，似乎和环境无关，因为有的是用 npx，有的用 pnpm dlx，但都挂了。而除了这 4 个以外剩下的所有都能正常运行。这就奇怪了。

### 3. 技术解释：单点深入

不解释所有相关概念。从问题链中选一个最关键的因果环节，讲清“为什么”和“所以怎样”。

**对比示例**：

> 原文：Node 16 缺少 `GlobalRequest`、`ReadableStream` 等现代 Web API，导致依赖这些 API 的 ESM 服务在启动时直接崩溃。
`@hono/node-server` 中 `class Request extends GlobalRequest` 拿到 `undefined`...
>
> 改写：Node 16 下启动会发现报错 `Class extends value undefined is not a constructor or null`。换 Node 22 就好了。根因就是 Node 版本。

### 4. 小标题：疑问句/短语推进

用问句或短句制造推进感，避免文档式标题。

**推荐形式**：
- `## 现象`
- `## 是 xx 问题吗？`
- `## 怎么修改呢？`
- `## 隐藏机制：xxx`
- `## 最终方案：xxx`

**避免**：
- `## 环境`
- `## 验证猜想`
- `## 原因分析`
- `## 总结`

### 5. 代码/数据：只留关键帧

保留三类片段：报错/异常、失败验证、成功验证。删除环境检查、中间推导、完整输出。

### 6. 结尾：轻量收尾

不需要总结段落。用一句验证成功的话或短句情绪收尾。

**示例**：
> OK，搞定了。

## 常见改写模式

| 原文特征 | 改写方向 |
|---------|---------|
| 长段落环境交代 | 压缩为 1-2 句个人背景 |
| 多原因分析 | 只保留一个核心原因 |
| 完整命令输出 | 只保留关键报错或成功提示 |
| 报告体过渡语 | 删除，改用断句或口语连接 |
| “对照实验”/“验证猜想” | 改为“验证过程非常简单，直接...” |
| 文末总结段落 | 删除，用一句轻量收尾 |

## 雷区

- 改得像教程或官方文档
- 每个概念都解释一遍
- 删除所有口语化表达
- 添加“综上所述”“总结一下”
- 保留过多环境铺垫和完整输出
- 小标题用文档式词汇
