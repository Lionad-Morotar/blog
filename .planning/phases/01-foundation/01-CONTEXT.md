<decisions>
## Implementation Decisions

### 四层认知结构的判断标准

#### 领域 (Domain)
- 按照 `0.index.md` 中的 20+ 分类作为领域定义
- 置信度一般，可能需要调整
- 每个领域对应一个以 `_` 为前缀的目录（如 `_frontend`）

#### 子领域 (Subdomain)
- **统一使用目录形式**，不按内容量区分文件或目录
- 每个子领域是一个子目录（如 `_frontend/css/`）
- 子领域目录内包含该子领域的入口文件（`0.index.md` 或 `{subdomain}.md`）

#### 主题 (Topic)
- 界定标准：**优先按照讨论范围**
- 如果主题有扩展潜力（语义上独立）则拆分成单独文件
- 或者：内容超过 150+ 行时拆分
- 一个主题对应一个 Markdown 文件

#### 知识点 (Knowledge Point)
- 界定：**观点、案例、洞见** 等思考性内容
- 保持为四级标题（`#### 知识点`）
- 不拆分为独立文件，避免过度碎片化
- 对于知识点是否应该拆分的边界情况，Claude 有灵活选择的权限

#### 现有文档的三级标题处理
- **重新审视每个三级标题**，决定其所属层级
- 符合主题级别 → 拆分成独立文件
- 符合知识点级别 → 降级为四级标题
- 保持当前层级 → 维持现状

### 链接保留策略

#### 需要保留的链接
- **所有内容链接** 都必须保留
- 包括 6.maps/ 内部的引用，以及其他目录对 maps 的引用
- 当前项目没有失效链接（Nuxt 构建会检测）

#### 链接更新机制
- **不保留旧 URL**，不采用重定向方案
- 移动文件时，**同步更新所有引用该文件的链接**
- 链接更新策略：Claude 根据情况选择最佳方法（自动化脚本或手动更新）
- 每次修改或创建文件后，需要验证链接有效性

### 重定向实现方式

#### 决策：不采用重定向
- 不使用 Nuxt 路由规则（`routeRules`）
- 不使用中间件重定向
- 不使用服务器路由重定向
- **采用直接修改链接地址的方式**

#### 原因
- 简化架构，避免维护两套 URL 系统
- 降低长期技术债务
- 要求每次移动文件时都必须正确更新所有引用

### 迁移粒度控制

#### 批次大小
- **按原计划执行：18 个阶段**
- 简单领域：每批次 4-5 个领域（分成 4 个批次）
- 中等领域：每批次 4 个领域（分成 5 个批次）
- 复杂领域：每个领域单独一个批次（7 个批次）
- 验证与集成：单独一个批次

#### 验证策略
- **Claude 决定最佳验证时机**
- 建议在每个阶段（批次）完成后进行验证，确保链接有效性
- 在复杂领域重构前，需要确认简单/中等领域的模式已稳定

</decisions>

## Specific Ideas

### 四层结构示例

以 `_frontend` 为例：

```
_frontend/                          # 领域（Domain）
├── frontend.md                     # 领域入口文件
├── typescript/                     # 子领域 - 目录形式
│   ├── typescript.md               # 子领域入口
│   └── 0.index.md                  # 导航文件
├── css/                            # 子领域
│   ├── css.md                      # 子领域入口
│   ├── layers.md                   # 主题（Topic）
│   ├── sprite-animation.md         # 主题
│   └── ...
└── w3c/                            # 子领域
    └── ...
```

### 标题层级处理原则

- 一级标题（`#`）：文件标题，通常是主题名称
- 二级标题（`##`）：主题的子分类或章节
- 三级标题（`###`）：可能是子主题（需要判断是否提升为文件）
- 四级标题（`####`）：知识点（观点、案例、洞见）

</specifics>
