# Maps 知识库重构

## What This Is

将 `content/6.maps/` 目录下的知识笔记从当前的扁平/混合结构，重构为统一的四层认知结构：领域 → 子领域 → 主题 → 知识点。每个文档和标题都需重新审视其在认知层级中的位置，并拆分/重组到正确层级。

## Core Value

知识库具备清晰的认知导航结构，读者可以按领域→子领域→主题→知识点的路径快速定位信息，作者可以明确知道新内容应该归档到哪里。

## Requirements

### Validated

(从现有 codebase 推断)
- ✓ 知识地图已按领域分类列出 20+ 个领域 — 现有
- ✓ 部分内容已使用子目录组织（如 `_frontend/css/`）— 现有
- ✓ 存在领域入口文件模式（`0.index.md`）— 现有

### Active

- [ ] 制定四层认知结构的判断标准（怎样的标题属于领域/子领域/主题/知识点）
- [ ] 完成所有 20+ 个领域的复杂度评估和阶段划分
- [ ] 按阶段逐个重构领域文档：
  - [ ] 阶段 1：结构简单的领域（1-3 个文件的领域）
  - [ ] 阶段 2：中等复杂度领域（4-8 个文件/子目录的领域）
  - [ ] 阶段 3：复杂领域（9+ 个文件或深层嵌套结构的领域）
- [ ] 为每个领域创建/更新入口文件（领域.md），包含子领域导航
- [ ] 重新审视并拆分现有三级标题到正确的认知层级
- [ ] 确保知识点（观点、案例、洞见）以四级标题形式存在
- [ ] 更新 `0.index.md` 中的链接指向新结构
- [ ] 验证重构后所有链接可访问

### Out of Scope

- 不修改内容本身，只调整结构 — 纯重构任务，不涉及内容更新
- 不迁移其他目录（如 2.articles、4.books）的内容 — 仅聚焦 6.maps
- 不添加新内容 — 仅重组已有内容
- 不改变 URL 路由规则 — 使用现有 Nuxt Content 路由机制

## Context

**当前状态**：
- Maps 目录下有 60+ 个子目录（`_frontend`、`_database`、`_ai` 等）
- 文件组织混合：有的领域是扁平的（如 `_go/go.md`），有的已有嵌套（如 `_frontend/css/`）
- `0.index.md` 中定义了 20+ 个领域的分类，但目录结构与分类不完全对应
- 部分长文档包含大量三级标题，需要重新审视其认知层级

**技术环境**：
- Nuxt Content v3 管理 Markdown 内容
- 文件路径决定 URL 路由
- 使用 `_dir.yml` 和 `0.index.md` 模式作为目录入口

**参考示例**：
```
_frontend/                          # 领域
├── frontend.md                     # 领域入口
├── typescript.md                   # 子领域（单文件）
├── css/                            # 子领域（目录）
│   ├── css.md                      # 子领域入口
│   ├── layers.md                   # 主题
│   └── sprite-animation.md         # 主题
└── w3c.md                          # 子领域（合并 w3c 相关内容）
```

## Constraints

- **技术栈**: Nuxt Content v3，Markdown + YAML frontmatter — 必须兼容现有系统
- **URL 兼容性**: 重构后尽量保持原有链接可用，或使用 Nuxt 重定向 — 避免死链
- **阶段执行**: 每次只重构一个领域，验证通过后再进行下一个 — 降低风险
- **四级标题保留**: 知识点层级保持为四级标题 `#### 知识点`，不作为单独文件 — 保持粒度适中

## Key Decisions
