---
title: UI Ralph Loop
description: 网页开发工作流：截图对比设计稿 → 多模态模型验证 → 修复代码循环
---

年前做了几个营销网站，都是用 Plan then Build 模式写的代码。因为那会儿 Gemini 的多模态能力强一些，所以用 GPT 做规划，用 Gemini 来写前端。

实际体验下来，创建页面很轻松，但 UI 对齐非常麻烦。光靠设计图，LLM 无法理解如何把阴影、圆角等具体样式转换成项目的设计语言。

后来给 AI 喂项目的组件文档和蓝湖的设计稿转的代码，勉强解决了 UI 对齐困难的问题。只要能接受 token 燃烧这种缺点，再加上一些人力投入，能明显感觉流程通畅了一些。

那会儿我琢磨着，能不能在项目中做一个「Playwright 截图对比设计稿 → 多模态模型验证 → 修复代码」的循环验证流程，但当时苦于研究 GSD 和各种 Skill，把这事儿忘了。

今天看到 [SWE-bench February 2026 leaderboard update](https://simonwillison.net/2026/Feb/19/swe-bench/) 这篇文章，里面提到一个通过 Playwright 注入 JS 代码，改变既有网站图表渲染的例子。

我尝试用不同的模型复现这个例子，效果都还不错，突然萌生了把 ui-ralph-loop 完成的想法。

恰好放假这会儿尝试了最近发布的新模型，感觉 Kimi K2.5 的多模态性能、成本、速度等综合能力相当不错，比新年发布的 Doubao 和 GLM5 等更出色，暂时就先选用 Kimi K2.5。
