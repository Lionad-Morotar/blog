# Top 10 Vibe Coding Secrets

## 1. 思维的原理

最重要的一点放在开头，你必须了解 LLM 是如何“思考”的，这样有助于你理解为什么要做X而不是Y才能提升 Vibe Coding 结果的质量。

我相信你知道LM是基于前一个词来预测下一个词的生成模型，但只是不理解LLM的此类基于概率的生成能达到思考的效果，以及通过思考提高结果的质量。我将用一个非常简单的例子向你解释“LLM 是基于概率的模型”。

在一个经典提示场景中，我询问“strawberry单词中有几个r”，模型直接告诉我一个数字“3”，这很好，但并不是所有情况下他都会告诉我数字3。嗯，在很多其他情况下，他会告诉我是2，是4，甚至是1。

回答一个数字会不准确的原因是因为LLM需要基于问题以及他的初始生成的内容去预测它下一个生成的内容，推理模型只是把这个步骤稍稍进行了扩展，使生成路径从单纯的数字变成了“1，等等...”。

我们知道“等等”意味着事情发生了转机，对于生成模型来说显然也是一样的，等等后面可以接很多描述，比如“等等，让我来验证这个数字的正确性...”，“等等，1 是错的”。

尽管在技术上不正确，但是从思维上把大模型的思考理解成在高维空间搜索理的裁剪似乎有助于对提高搜索质量的认识。在接下来几个很实际的秘密中，更能体现这一点的作用。

## 2. 精细化你的提示词

你需要尤其注意两点：流程以及细节。

流程意味着你对1234,以及 1.1, 1.2 这些步骤有着清晰的了解。你用精心设计的步骤收缩了大模型的搜索流搜索过程，让它在后续的生成中发挥更好的效果。

细节意味着你把一些极度琐碎但确是关键要点的地方告诉了大模型，在这些关键要点大模型，它可能会通过反复试错的方法尝试出答案，但一旦你把关键要点提前告知了他，意味着这也是意味着减少了大模型的搜索过程，可以让他在更顺利完成流程。

流程意味着大方向，细节意味着小方向，大方向和小方向都要管控，这似乎听起来有些矛盾。实际上，你需要通过自己的编码经验去平衡平衡这两点。

对于细节而言，“npm create nuxt project xxx” 要优于 “创建一个 nuxt 项目，目录名用 xxx”。

对于流程而言：

```
1. 需求澄清
   1.1 项目重述（Re-ground）
   1.2 研究目标分类
   1.3 模式专用提问（聚焦/扫描/全面）
   1.4 逃生舱口（Escape hatch）
   1.5 输出研究需求文档（YAML）
2. 评估任务构建查询
   2.1 确定搜索模式（easy/medium/hard）
   2.2 选择初始研究方向
   2.3 创建搜索任务（todos）
   2.4 构建查询集群（运用高级搜索技巧）
3. 分发子代理搜索（并行）
   3.1 创建搜索团队（TeamCreate）
   3.2 启动并行搜索（2~4 个 Agent）
   3.3 子代理写入独立文件（turn-{n}/search-agent-{n}-task.md）
4. 聚合结果到结构化日志
   4.1 初始化会话目录（/tmp/search-web/{date}/{task}/）
   4.2 创建协调日志（master-log.md）
   4.3 父代理记录事件时间线
5. 评估器 - 优化器循环
   5.1 启动评估器（读取 turn-{n}/search-agent-*.md）
   5.2 评估器输出报告（turn-{n}/search-evaluator-turn-{n}.md）
   5.3 决策点（全 5/5 则 COMPLETE，否则 CONTINUE）
   5.4 启动优化器（生成 turn-{n}/query-optimizer-turn-{n}.md）
6. 交付经过验证的结果
   6.1 生成最终报告（report.md）
   6.2 向用户展示主日志位置
```

要优于“通过三层Ask澄清需求并输出YAML文档，评估任务难度后构建查询集群，创建搜索团队并启动2~4个子代理并行搜索，聚合结果到结构化会话日志，经评估器-优化器循环迭代直至评分达标，最终生成验证报告并交付用户审核。”

人类语言拥有高度模糊的特征，但因为它是线性的，所以已经隐含了逻辑先后顺序。面向LM，我们需要把逻辑显示的通过结构化流程明确表达出来，而不是通过自然语言让LM进行猜测。

## 3. 给代理正确的上下文

上下文管理不仅仅是 Agent 框架需要做的事情，更是和你的提示词息息相关。我不是想拾起那些老生长谈的话题，而是因为 Context Engineering 怎么强调也不为过。

给大模型一个截图一段代码一个开源项目的网址，甚至是一个指向note models源码的目录，这些都算上下文，只要这些内容对大模型解决问题有帮助，那么就能收缩大模型运行时的搜索方向，有助于它提高运行效率和得到更高准确性的结果。

## 5. 武装到牙齿

有太多好玩的东西了是不是，这个小节我介绍一些我正在使用的插件或工具：

* 微信语音输入：微信语音输入要比whisper准确率高太多了，编程这种中英文混合的场景更是秒杀后者
* 双拼：我曾有肌腱炎，用双拼后改善了（我在瞎扯，因为主要原因可能是我不弹琴了）
* ccstatusline：给你的cloud定制展示内容，比如说正在使用的模型，已经使用的上下文的比例，筛选ID，然后token输出速度之类的
* claude-mem：号称能减少90%的token使用量，但实际我没测过
* SDD：选择一个好的SDD规范，且不论它对生成质量的影响，尽管不能加速你的开发流程，但可以显著降低思维负担，Superpowers、OpenSpec、BEAD、Speckit、GSD 太多了任君挑选
* RTK：节约特定场景的token使用量
* cmux：高性能可编程的终端界面，我喜欢使用它管理全局任务，而IDE则对应具体的项目开发
* 终端必备：zoxide、fabric、customized zshrc
* 系统和应用快捷键：给你10秒钟时间，使用任何工具，定位到你去年某个项目的package.json文件，查看是否有X依赖，如果没有完成，那么你失败了。

## 6. 核动力牛马

核动力牛马就是指那些24小时运行的系统，当然不是像open cloud这种，有丰富的定时任务组成的看起来看似连续运行的系统。核动力牛马我对核动力牛马的定义能停留在，嗯，就是氛围编码的基础上。

你如果用一个简短的提示词让大模型工作，那他可能会猜测一个模糊的目标。他有很多方式停下来，比如猜测了一个模糊的目标，然后丢失了具体的方向，然后忽然使用ask工具来问你，我需要选ABC哪哪几个方向。

另一种情况是因为任务细节不够明确，他选择了一条不是那么稳定的随有随机性制的路径，而不是那么稳定一定意味着执行结果质量不高。

所以最好的方式仍然是提高提示词的质量。但这并不是说我们要手动去优化提示词，而是意味着要用元提示的方式来拓展提示词的质量。

SDD 工具算元提示的一种，市面上有非常多类似的工具，这几个月我看见的新兴趣点是从SDD工具演正在向工作流的方式演变。

当然也不是说非得要完整的去学，用一套完整的去学习一套系统的工作流，你完全可以根据现有的SSSDD工具，去组合出一套属于自自己用着顺手的工作流程。

核心原理非常简单，还是plan+build，通过plan把任务详尽的拆分出来，再通过build精确执行，在此基础上，worktree、subAgent、team、ralph 等等概念都只是添彩。

前几周，我尝试组合流行的 SDD 工具，搭建了一套简单有效的工作流程：

1. `/grill-me`，从零散的想法组织并打磨提示词
2. `/office-hours`，用于项目启动，识别项目价值
3. `/plan-eng-review`，从工程师的视角审查架构、性能、测试等各方面的决策
4. `/gsd-new-milestone`，创建里程碑，并将需求完全的拆解到不同的阶段，把阶段拆解到不同的任务
5. `/gsd-discuss-phase [x]`，选择复杂阶段，并对具体实现做简单选择防止 Agent 跑偏
6. `/gsd-autonomous`，从0~1实现需求
7. `/ce:review`，派遣审计团队对现有实现进行详细到脚趾的核查
8. `/qa`，使用浏览器审查问题并解决

与人有关的大概是12345这5个步骤，然后有些步骤可以跳过，一般我会在晚上或或者说某个具体的时间点（其实就是那个5小时额度恢复的时候）设置一个定时任务，然后开始第6步或者第7步，第8步。

这套工作流的好处是，在前五个步骤把需求彻底聊清楚之后，在代码实践部分，它可以做到完全的自动化。

前几周，在我的实践，不开团队，不开ralph，编码实现步骤连续跑了 4个半小时 96 个提交。

而且这个自动化不是简单的像这个视频网站，像有些人开启了一个这个team团队，然后扔一个简单的提示词做个营销网站，它就能吭哧吭哧跑半天，而是基于你聊的需求，做一个完整的至少是工具级别甚至是生产级别应用。

这还是在工作流没有精心打磨的情况下，我相信往后几个月SDD会逐渐熄火，编码热情会逐渐演变成各种工程化流程或者说工作流满天飞的情况。

## 8. 提高代码能力

尽管普遍的认识是 Agent 编程能力飞速提升，成为廉价问题的最佳性价比解决方案，会抹杀初级工程师入门途径。

但在广泛的论坛讨论中，我们会更倾向于看到人们的实际谈论的是，Agent 在帮助初级工程师完成飞速的到中级工程师的提升，放大高级工程师的能力成为10倍程序员，而中级工程师卡在因为缺失了多项维度能力导致的不能快速成长的尴尬地步。

使用氛围编码能提高你使用cloud之类的工具的能力，但不会提高你的代码能力，如果你不想变得那么急功近利，且依旧热爱编码，那么静下心来手工解决一个问题，是一种新式的不错的冥想方式。

而单点氛围编码技能的重要性会随着企业级重型工作流的普及逐渐降低，这之后，如何平衡架构以及技术细节的掌握能力，会成为开发人员的职业发展的长期目标。
